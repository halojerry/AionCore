# OPC 验证闭环

OPC 最需要的是“小而快的验证”，不是完整系统。

## 三层时间盒

### 72 小时

验证一个最关键假设。

例：
- 找 5 个目标用户聊 20 分钟
- 发一个公开招募帖
- 做一页服务说明页
- 发起一次预售
- 手工跑一次服务流程

### 7 天

验证用户是否愿意持续投入。

例：
- 3 次访谈 + 1 次付费试单
- 1 个最小手工服务流程
- 3 篇内容测试同一个主张
- 1 个 Agent 工作流跑通内部流程

### 14 天

验证是否值得产品化。

例：
- 至少 3 个非人情用户愿意付费 / 预约 / 转介绍
- 重复流程能否抽象成模板
- 用户是否在第二次主动回来

## 停止线模板

```markdown
如果 {N 天内} 没有达到 {X 个强信号}，就 {停止 / 缩小范围 / 换目标用户 / 改问题定义}。
```

强信号通常包括：付费、预订、转介绍、主动追问、愿意给真实材料。  
弱信号包括：点赞、说“有意思”、泛泛鼓励。

## OPC 原则

先手工，后自动。  
先验证，后产品化。  
先有判断，再让 Agent 放大执行。
