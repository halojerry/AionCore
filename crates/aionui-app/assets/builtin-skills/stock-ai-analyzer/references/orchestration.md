# 工程编排与上下文管理

> 按需加载。当任务涉及多步取数、读原文、联网（尤其 Deep 模式 + 成长成功率方向三）时，主 agent **先读本文定编排，再动手**。

## 0. 两个杀手 + 三条铁律

Deep 模式失败几乎只有两种死法：

- **上下文爆炸**：把 PDF 全文 / 公告原文 / 全量 evidence 直接读进主上下文，没读完几篇窗口就满了，任务半截死掉、写不出报告。
- **外联失活**：大量 `fetch` / 联网里有几个失败或超时，流程卡死、空转，或干脆缺数据硬编。

两者都**不是分析能力问题，是编排问题**。三条铁律，违反任一，Deep 基本不可能在一次上下文内完成：

1. **主 agent 不碰大文本**：PDF 全文、公告原文、长 JSON 只在子代理 / 取数脚本里消费；回主上下文的**永远只有结构化结论块**。主 agent 的上下文 = 分型 + scan 摘要 + 各子任务结论 + 最终整合，仅此。
2. **计划先行**：动手前先把要做的子任务列成**工单清单**（§2），带预算和落点；执行中逐项核销。没清单就开干 = 必然漏项或超预算。
3. **子任务隔离**：每个独立调研项派一个子代理，给定"读哪篇 / 查什么 / 透镜 / 预算 / 固定回报模板"，主上下文只收回报。无子代理能力的 agent 降级为"读完即焚"的串行（§3）。

## 1. 上下文预算：怎么保证"一次窗口内完成"

- **证据分层**（pack 三产物）：`evidence` 归档、不整体入主上下文；`analysis_context` 轻量、主 agent 可吃；`module_*.json` 分发给子代理。
- **主上下文线性可控**：N 个子任务 → N 个结论块，每块只有固定模板几行。列完工单就能估出主上下文峰值 ≈ 项数 × 每项回报量。
- **原文即焚**：子代理读完 PDF / 公告，抽出引文 + 章节 / 页 + 一句判断，**立刻丢弃全文**，绝不回传主 agent。
- **大 JSON 也分发**：不在主 agent 里展开完整 evidence 或历年报表；要某段就让子任务去取那一段。
- 估算后若明显超窗口 → **缩范围**（按优先级砍低价值项，§4 停止条件），不要硬塞到爆。

## 2. 计划先行：工单清单协议

动手前产出一份**调研工单**（research ledger）。Deep 模式必做；骨架完整单股研究建议做（`pack → module 子代理`本就是隐式工单）。

工单每项的字段：

| 字段 | 说明 |
|---|---|
| 编号 | W1 / W2 … |
| 归属 | 命题 / 旗标 / 维度（A/B/C/D）/ 模块 |
| 动作 | 读哪篇 + 哪段 / 查什么关键词 / 哪个 `fetch` |
| 预算 | 该项允许的读 PDF / 联网次数 |
| 产出 | 期望回报（结论块）+ 落点章节 |
| 状态 | ⬜ 待办 / ✅ 完成 / ⚠️ 数据缺（降置信度）/ ✗ 超预算放弃 |

**起点自动生成**：`python scripts/thesis_scan.py --evidence <pack> --plan` 会把 `thesis_candidates` + `anomaly_flags` + `people_governance.probes` 去重、编号、配预算，吐一份**工单骨架**（markdown）。主 agent 在此之上增删项、定优先级、补落点——脚本只摆，不定优先级判断。

> 有 TodoWrite / 任务工具的 agent 用工具维护这份清单；没有的用 markdown 表，状态列手动核销。两者等价，关键是**有一份可核销的完整清单**。

执行 = 逐项领取 → 派子代理 → 收结论块 → 核销。**全部核销且高优先项已覆盖 = 可收口**。

## 3. 子代理编排

- **合约**：主 agent 写给子代理的指令必须含——分型透镜（"按 X 型口径分析"）、要读的具体文件与章节、要回答的问题、预算、**固定回报模板**（`deep_mode.md §6` 的五字段块）。缺模板字段就退回重发，避免拿到松散文字最后整段丢掉。
- **隔离**：一项一代理；读 PDF / 公告原文的子代理尤其要隔离，全文不出子上下文。
- **并行 fan-out**：相互独立的调研项（不同旗标、不同公告、不同主线词）**并行派发**，主 agent 同一轮收多个结论块——省时，且中间过程不污染主上下文。只有有依赖的（A 的结论决定 B 查什么）才串行。
- **降级（无子代理能力的 agent）**：按工单顺序**串行**——取一项的原文 → 抽结论写进笔记 → **丢弃原文再做下一项**，绝不累积多篇全文。聚合阶段只读各项结论 + `analysis_context`。

## 4. 外联健壮性

- **分类**：内联原子取数（`data_fetcher`，确定性、自带 `retry.py` + `cache.py`）vs 外联（联网检索、PDF 下载）。
- **失败降级矩阵**：取数失败 → 重试（retry 已内置）→ 命中缓存（cache）→ 仍失败则该项标 **⚠️ 数据缺**、结论里写"未取得"、对应**降置信度**。绝不空转重试到死，绝不编造填补。
- **幂等 / 缓存**：重复取数走缓存，省外联配额；同一票多轮分析不重复下载同一 PDF。
- **预算与停止**：每项工单带预算；总预算见 `deep_mode.md §6`（默认读 ≤10 篇 / 联网 ≤10 次）。命题已证实 / 证伪、或边际信息很低即停，不为补全而补全。

## 5. 单次完成自检（写报告前过一遍）

- 工单是否**全部核销**（✅/⚠️/✗ 三态齐全，无 ⬜ 遗留）？
- 主上下文有没有被原文污染（是否误把 PDF 全文 / 完整 evidence 读进了主 agent）？
- 高优先命题 + high 旗标是否都有结论块？
- 超预算的项是否**显式放弃并降置信度**，而不是半截挂着？
- 每条结论块能否直接落进报告对应章节（Deep 还要进「深度调研索引」）？

任一不过 → 回去补，不提前结案。

## 6. 与现有协议的衔接

- `[Deep 模式启动]` 协议 = 工单的"摘要版"对外播报；工单本体是内部执行清单。
- 每个 ✅ 工单项 → 一条 `==深度调研发现==`（就近嵌正文）→ 「深度调研索引」一行。
- ⚠️ 数据缺 / ✗ 放弃 的项 → **也在索引登记一行并注明状态**（不嵌正文），并汇入估值结论的**置信度下调依据**（§5.6）。
- **完整性闭环**：索引行数 == 已执行工单项数。正文 + 索引覆盖整张工单——查过的东西无一遗漏地体现，这是 Deep 模式"查多少体现多少"的硬约束。
